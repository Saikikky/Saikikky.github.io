Java的异常体系

###Error和Exception的区别

  • Error：程序无法处理的系统错误，编译器不做检查(比如JVM的问题，如系统崩溃、内存空间不足、虚拟机错误、方法调用栈溢出等)
    • NoClassDefFoundError —— 找不到class定义的异常
    • StackOverflowError —— 深递归导致栈耗尽
    • OutOfMemoryError —— 内存溢出异常
  • Exception：程序可以处理的异常，捕获后可能恢复
    • RuntimeException：不可预知的，程序应当自行避免。(数组下标越界、访问空指针等)
      • NullPointerException —— 空指针引用异常
      • ClassCastException —— 类型强制转换异常
      • IllegalArgumentException —— 传递非法参数异常
      • IndexOutOfBoundsException —— 下标越界异常
      • NumberFormatException —— 数字格式异常
    • 非RuntimeException：可预知的，从编译器校验的异常。(IOException等)
      • ClassNotFoundException —— 找不到指定class的异常
      • IOException —— IO操作异常

finally会在catch的return语句之前执行

Java异常处理消耗性能的地方

  • try-catch块影响JVM的优化
  • 异常对象实例需要保存栈快照细信息，开销巨大

1.Java内存结构=JVM内存结构

程序计数器

  • 当前线程所执行的字节码行号指示器(逻辑)
  • 改变计数器的值来选取下一条需要执行的字节码指令
  • 和线程是一对一的私有关系
  • 对Java方法计数(记录的是正在执行的虚拟机字节码指令的地址)；Native方法，则为Undefined
  • 不会发生内存泄漏

Java虚拟机栈

虚拟机栈描述的是Java方法执行的内存模型：每个方法在执行的同时都会创建一个栈帧用于存储局部变量表、操作数栈、动态链接、方法出口等信息。每一个方法从调用直到执行完成的过程，就对应一个栈帧在虚拟机栈中入栈道出栈的过程。

  • 局部变量表：包含方法执行过程中的所有变量

  • 操作数栈(在执行字节码指令过程中会用到)：入栈、出栈、复制、交换、产生消费变量

本地方法栈

  • 与虚拟机栈类似，主要作用于标注了native的方法。

Java堆(Heap)

  • 对象实例的分配区域
  • GC管理的主要区域

方法区

用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据。和Java堆一样不需要连续的内存和可以选择固定大小或者可扩展外，还可以选择不实现垃圾收集。这区域的内存回收目标主要是针对常量池的回收和对类型的卸载。当方法区无法满足内存分配需求时，将抛出OutOfMemeoryError异常。

元空间(没有字符串常量池移入堆中)与永久代：都是用来存储class的相关信息，均是方法区的实现。

  • 元空间使用本地内存，而永久代使用的是jvm的内存
  • 字符串常量池存在永久代中，容易出现性能问题和内存溢出
  • 类和方法的信息大小难确定，给永久代的大小指定带来困难
  • 永久代会给GC带来不必要的复杂性

2.JVM调优

JVM三大调优参数-Xms -Xmx -Xss

  • -Xss：规定了每个线程虚拟机栈(堆栈)的大小 256k足够 会影响此进程中并发线程数的大小
  • -Xms：堆的初始值
  • -Xmx：堆能达到的最大值

Java内存中的堆和栈的区别——内存分配策略

  • 静态存储：编译时确定每个数据目标在运行时的存储空间需求
  • 栈式存储：数据区需求在编译时未知，运行时模块入口前确定
  • 堆式存储：编译时或运行时模块入口都无法确定，动态分配

联系：引用对象、数组时，栈里定义变量保存堆中目标的首地址

区别：

  1. 管理方式：栈自动释放，堆需要GC
  2. 空间大小：栈比堆小
  3. 碎片相关：栈产生的碎片远小于堆
  4. 分配方式：栈支持静态和动态分配，而堆是动态分配
  5. 效率：栈的效率比堆(双向链表)高

3.JVM如何加载.class文件

JVM架构

JVM主要分为四个部分：Class Loader、Runtime Data Area、Execution Engine、Native Interface

Class Loader：依据特定格式，加载class文件到内存。

Execution Engine：对命令进行解析(class文件的字节码)，并提交给操作系统去执行。

Native Interface：融合不同开发语言的原生库为Java所用(native方法)

Runtime Data Are：JVM内存空间结构模型。

类从编译到执行的过程

  • 编译器将.java源文件编译成.class字节码文件
  • ClassLoader将字节码转换为JVM中的Class<…>对象
  • JVM利用Class<…>对象实例化为…对象

ClassLoader

ClassLoader主要工作在Class装载的加载阶段，主要作用是从系统外部获得Class二进制数据流。它是JVM的核心组件，所有的Class都是由ClassLoader加载的，ClassLoader负责通过将Class文件里的二进制数据流装载进系统，然后交给Java虚拟机进行连接、初始化操作。

ClassLoader的种类

  • Bootstrap ClassLoader(启动类加载器)：C++编写，加载核心库java.*
  • ExtClassLoader(扩展类加载器)：Java编写，加载扩展库javax.*
  • AppClassLoader(应用程序类加载器)：Java编写，加载程序所在目录class path
  • 自定义ClassLoader：Java编写，定制化加载
双亲委派

​ 如果一个类加载器收到了类加载的请求，它首先不会自己去加载这个类，而是把这个请求委派给父类加载器去完成，每一个层次的类加载器都是如此，因此所有的加载请求最终都应该传送到顶层的启动类加载器中，只有当父加载器反馈自己无法完成这个加载请求(搜索范围中没有找到所需的类)时，子加载器才会尝试自己去加载。 自底向上再自顶向下

  • 为什么要使用双亲委派机制去加载类

    • 避免多份同样字节码的加载

      例如类java.lang.Object，存放在rt.jar中，无论哪一加载器加载，最终都委派给了处于模型最顶端的启动类加载器进行加载，因此Object类在程序的各种类加载器环境中都是同一个类。

类的加载方式

  • 隐式加载：new
  • 显式加载：loadClass、forName等。获取对象之后需要调用getInstance方法获取对象的实例

类的加载过程(Class对象的生成过程用以区分)

加载

通过ClassLoader加载class文件字节码，生成Class对象。

  • 通过一个类的全限定名来获取定义此类的二进制字节流
  • 将这个字节流所代表的静态存储结构转化为方法去的运行时数据结构
  • 在内存中生成一个代表这个类的java.lang.Class对象，作为方法区这个类的各种数据的访问入口
验证
  • 检查加载的class的正确性和安全性
准备
  • 为类变量分配存储空间并设置类变量初始值(通常为零值，但final修饰直接给值)
解析
  • JVM将常量池内的符号引用转换为直接引用
初始化
  • 执行类变量赋值和静态代码块

loadClass和forName的区别

  • Class.forName得到的Class是已经初始化完成的。(即会执行静态代码块等)
  • ClassLoader.loadClass得到的class是还没有链接(验证、准备、解析)的

对于一些需要执行静态方法的比如数据库驱动加载Class.forName("com.mysql.jdbc.Driver")

在SpringIOC中，在资源加载器获取要读入的资源时，在读取一些bean文件时，如果是以class path的方式加载，就需要使用ClassLoader.loadClass方法加载，跟SpringIOC的lazy loading(延迟加载)有关，SpringIOC为了加快初始化速度，大量使用延时加载，classLoader不需要执行类中的初始化步骤也不需要链接，可以加快加载速度，而初始化工作等到最后要使用到类的时候再去做。

4.GC

判断对象是否为垃圾的算法

引用计数算法

  • 通过判断对象的引用数量来决定对象是否可以被回收
  • 每个对象实例都有一个引用计数器，被引用则+1，完成引用则-1
  • 任何引用计数为0的对象实例可以被当作垃圾收集

优点：执行效率高，程序执行受影响较小

缺点：无法检测出循环引用的情况，导致内存泄漏

可达性分析算法(用的比较多)

算法基本思想是：通过一系列的成为“GC Roots”的对象作为起始点，从这些节点开始向下搜索，搜索所走过的路径称为引用链，当一个对象道GC Roots没有任何引用链相连，即从GC Roots到这个对象不可达，则证明此对对象是不可用的。就可以被判定为是可以被回收的对象。

Java中可作为GC Roots的对象包括一下几种

  • 虚拟机栈(栈帧中的本地变量表)中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中JNI(即一般说的Native方法)引用的对象
  • 活跃线程的引用对象

垃圾收集算法

标记-清除算法

由于碎片问题用的较少

复制算法

适用于存活率低的区域(新生代)

标记-整理算法

适用于存活率高的区域(老年代)

分代收集算法

  • Minor GC

    发生在新生代，通常会使用复制算法

    新生代：尽可能快速地收集掉那些生命周期短的对象

    Eden区：survivor区(from)：survivor区(to)=8:1:1

    ​ 每次使用Eden区和其中一块survivor区，回收时，讲Eden和Survivor中还存活着的对象一次性复制到另一个Survivor空间上，最后清理掉Eden和刚才用过的survivor空间。当survivor空间不够用时，需要依赖其他内存(老年代)进行分配担保。

    Eden空间不足会触发Minor GC

  • Full GC

    老年代的GC通常会伴随着新生代的GC。标记清除和标记整理算法

    新生代的对象可以晋升到老年代

    • 经历一定Minor次数依然存活的对象(默认15次)
    • survivor区中存放不下的对象
    • 新生成的大对象

    老年代：存放生命周期较长的对象

    Full GC比Minor GC慢，但执行频率低

    老年代空间不足，JDK1.7以前永久代空间不足，Minor GC晋升到老年代的平均大小大于老年代的剩余空间，调用System.gc()会触发Full GC

  • Stop The World

    • JVM由于要执行GC而停止了应用程序的执行
    • 任何一种GC算法都会发生
    • 多数GC优化通过减少Stop The World发生的时间来提高程序性能

  • safepoint 安全点

    程序执行时并非在所有地方都能停顿下来开始GC，只有在到达安全点时才能暂停

    • 分析过程中对象引用关系不会发生变化的点
    • 产生safepoint的地方：方法调用、循环跳转、异常跳转等
    • 安全点数量要适中

垃圾收集器

Serial收集器(-XX:+UseSerialGC, 复制算法)

  • 单线程收集，进行垃圾收集时，必须暂停所有工作线程
  • 简单高效，Client模式下默认的年轻代收集器(JVM运行模式分为Server和Client)

ParNew收集器(-XX:+UseParNewGC,复制算法)

  • 多线程收集，其余的行为、特点和Serial收集器一样
  • 单核执行效率不如Serial，在多核下执行才有优势

Parallel Scavenge收集器(-XX:+UseParallelGC,复制算法)

吞吐量=运行用户代码时间/(运行用户代码时间+垃圾收集时间)

  • 比起关注用户线程停顿时间，更关注系统的吞吐量
  • 在多核下执行才有优势，Server模式下默认的年轻代收集器

Serial Old收集器(-XX:+UseSerialOldGC,标记-整理算法)

  • 单线程收集，进行垃圾收集时，必须暂停所有工作线程
  • 简单高效，Client模式下默认的老年代收集器

Parallel Old收集器(-XX:+UseParallelOldGC,标记-整理算法)

  • 多线程，吞吐量优先

CMS收集器(-XX:+UseConcMarkSweepGC,标记-清除算法)

  • 初始标记：Stop The World，仅仅只标记一下GC Roots能直接关联到的对象，速度很快。
  • 并发标记：并发追溯标记，程序不会停顿。进行GC Roots Tracing
  • 并发预清理：查找执行并发标记阶段从年轻代晋升到老年代的对象，减少下一阶段工作量(书中没有这一阶段)
  • 重新标记：Stop The World，暂停虚拟机，扫描CMS堆中的剩余对象。为了修正并发标记期间因用户程序运作而导致标记产生变动的那一部分对象的标记记录。这个阶段的停顿时间会比初始标记阶段时间稍微长一些，但远比并发标记的时间短。
  • 并发清除：清理垃圾对象，程序不会停顿
  • 并发重置：重置CMS收集器的数据结构(书中没有这一阶段)

G1收集器(-XX:+UseG1GC,复制+标记-整理算法)

  • 并行和并发
  • 分代收集
  • 空间整合
  • 可预测的停顿

Java中的强引用、软引用、弱引用、虚引用

强引用(Strong Reference)

  • 最普遍的引用：Object obj = new Object()
  • 抛出OutOfMemoryError终止程序也不会回收具有强引用的对象
  • 通过将对象设置为null来弱化引用，使其被回收

软引用(Soft Reference)

  • 对象处在有用但非必须的状态
  • 只有当内存空间不足时，GC会回收该引用的对象的内存
  • 可以用来实现高速缓存

弱引用(Weak Reference)

  • 非必须的对象，比软引用更弱一些
  • GC时会被回收
  • 被回收的概率也不大，因为GC线程优先级比较低
  • 适用于引用偶尔被使用且不影响垃圾回收的对象

虚引用(PhantomReference)

  • 不会决定对象的生命周期
  • 任何时候都可能被垃圾收集器回收
  • 跟踪对象被垃圾收集器回收的活动，起哨兵的作用
  • 必须和引用队列ReferenceQueue联合使用

1.MyISAM和InnoDB关于锁的区别

MyISAM：默认用的是表级锁，不支持行级锁。

表级锁和索引无关

myIsam下，进行select操作时，会自动加上一个表级的读锁，增删改会自动加上一个表级的写锁。

  • 当读锁未被释放时，如果有其他操作要进行写操作，会被阻碍直至所有的读锁都被释放。

  • 当写锁未释放时，读操作也会被阻碍直至所有的写锁都被释放。

  • 当写操作未释放时，写操作也会被阻碍直至所有的写锁都被释放。

读锁也叫共享锁，写锁也叫排他锁

手动加写锁：select语句后加上for update即将读锁变为排他锁，其他读操作也被阻碍直至排他锁释放。

InnoDB：默认的是行级锁，也支持表级锁。(支持事务)

除了用id、主键索引以外，只要索引涉及到的行都会被上共享锁或者排他锁。

在没有涉及到索引时，innodb用的是表级锁

默认事务是自动提交的，所以就会看起来并未加锁。

InnoDB是二段锁，即在一个事务操作中，分为加锁阶段和解锁阶段，且所有的加锁操作在解锁操作之前。默认行级锁，只对一行数据上锁，针对不同行数据时不会有冲突。

非阻塞select：InnoDB对select操作会有一个改进，会导致针对同一数据的select和update操作都可以成功。即没有对select上读锁。

手动加读锁：select语句后加上lock in share mode就是加上共享锁。

解锁：在一个事务中，只有在commit或者rollback时，才是解锁阶段。

除此之外，InnoDB还支持表级的意向锁，意向锁是一种不与行级锁冲突表级锁，意向锁分为意向共享锁IS(事务有意向对表中的某些行加共享锁)和意向排他锁IX(事务有意向对表中的某些行加排他锁)。意向锁是由数据引擎自己维护的，用户无法手动操作意向锁，在为数据行加共享 / 排他锁之前，InooDB 会先获取该数据行所在在数据表的对应意向锁。在进行标级别操作时，不用一行一行去轮询是否存在行级锁，直接查询意向锁就ok。意向锁与表级锁时互斥的，意向锁和意向锁是兼容的。意向锁在保证并发性的前提下，实现了行锁和表锁共存且满足事务隔离性的要求。

MyISAM和InnoDB适合的场景

  • MyISAM
    • 频繁执行全表count语句。(专门用一个变量保存了全表数据行数，InnoDB则没有需要每次重复操作)
    • 查询频繁，增删改比较少(存储碎片比较多，即使在尾部插入)。
    • 没有事务。
  • InnoDB
    • 数据增删改查比较频繁。
    • 可靠性要求比较高，要求支持事务。

数据库的锁分类

  • 按照锁的粒度划分：表级锁、行级锁、页级锁(相邻逻辑块)
  • 按照锁的级别划分：共享锁、排他锁
  • 按照加锁方式划分：自动锁、显式锁(语句加锁)
  • 按操作划分：DML锁(对数据进行操作上的锁..增删改查)、DDL锁(对表结构进行变更)
  • 按照使用方式划分：乐观锁(版本号、时间戳)、悲观锁

Checkpoint技术

InnoDB的内存结构中存在缓冲池，缓冲池的设计目的是为了协调CPU速度和磁盘速度，所以页的操作首先是在缓冲池中完成的，也就是说，如果有更新操作之后，页是脏的，即缓冲池中的页的版本要比磁盘中的新，Checkpoint机制将脏页刷新回磁盘。

Checkpoint目的：缩短数据库的恢复时间；缓冲池不够用时，将脏页刷新到磁盘；重做日志不可用时，刷新脏页。

  • 数据库只需对Checkpoint后的重做日志进行恢复，因为Checkpoint之前的页都已经刷新回磁盘。
  • 缓冲池不够用时，LRU算法溢出最近最少使用的页，若为脏页，需要强制执行Checkpoint，将脏页刷回磁盘。
  • 重做日志被重用的部分是指这些重做日志已经不再需要，当数据库发生宕机时，数据库恢复操作不需要这部分的重做日志，这部分就可以被覆盖重用。如果还需要使用就需要强制Checkpoint。

如果在从缓冲池将页的新版本刷新到磁盘时发生了宕机，那么数据就不能恢复了

为了避免数据丢失，当前事务数据库采用Write Ahead Log策略，当事务提交时，先写重做日志，再修改页。当发生宕机导致数据丢失，可以根据重做日志完成数据恢复。这也是事务ACID中D(持久性)的要求。

  • Sharp Checkpoint

    发生在数据库关闭时将所有的脏页都刷新回磁盘。(默认)

  • Fuzzy Checkpoint

    运行时，即引擎内部使用，只刷新部分脏页回磁盘。

InnoDB关键特性

插入缓冲(Insert Buffer)：性能提升

Insert Buffer和数据页一样，也是物理页的一个组成部分。应用程序中行记录的插入顺序是按照主键递增的顺序进行插入的。(主键索引是顺序的，不需要磁盘的随机读取)

对于非聚簇索引的插入或更新操作，不是每一次直接插入到索引页中，而是先判断插入的非聚集索引页是否在缓冲池中，若在，直接插入，若不在，先放入到一个Insert Buffer对象中，此时起到欺骗的效果，看似数据库这个非聚簇的索引已经插入到叶子节点中，而实际却没有，只是存放在另一个位置，然后再以一定的频率和情况进行Insert Buffer和辅助索引叶子节点的merge操作，这时通常能将多个插入合并到一个操作中，因为在一个索引页中，这就大大提高了对于非聚集索引插入的性能。

Insert Buffer的使用需要同时满足两个条件：

  • 索引是辅助索引
  • 索引不是唯一的

满足以上两个条件时，InnoDB存储引擎会使用Insert Buffer，这样就能提高插入操作的性能了。

例外情况 ：当应用程序大量的插入操作且都涉及到不唯一的非聚集索引时，也就是使用了Insert Buffer，若此时数据库发生宕机，大量的Insert Buffer还没有合并到实际的非聚簇索引中，这个情况下的数据库恢复可能需要很长时间。

辅助索引不能是唯一的，因为在插入缓冲时，数据库并不去查找索引页来判断插入的记录的唯一性，如果去查查哦肯定又会有离散读取的情况发生，从而导致Insert Buffer失去了意义。

Insert Buffer存在一个问题，在写密集的情况下，插入缓冲会占用过多的缓冲池内存(innodb_buffer_pool)，默认最大可以占用到1/2的缓冲池内存，修改IBUF_POOL_SIZE_PER_MAX_SIZE就可以对插入缓冲的大小进行控制，比如将IBUF_POOL_SIZE_PER_MAX_SIZE改为3，则表示最大只能使用1/3的缓冲池内存。

Change Buffer (Insert Buffer升级版)

适用对象依然是非唯一的辅助索引。

InnoDB存储引擎可以对DML操作——Insert、Delete、Update都进行缓冲，分别是Insert Buffer、Delete Buffer、Purge Buffer。对一条记录进行Update操作可能分成两个过程：将记录标记为删除；再真正的将记录删除。Delete Buffer对应Update操作的第一个过程，Purge Buffer对应Update操作的第二个过程。

Insert Buffer的内部实现

Insert Buffer的数据结构是一棵B+树，现在的MySQL版本中，全局只有一棵Insert Buffer B+树，负责对所有的表的辅助索引进行Insert Buffer。而这棵B+存放在共享表空间中，即ibdata1中。因此试图通过独立表空间ibd文件恢复表中数据时，往往会导致CHECK TABLE失败。因为表的辅助索引中的数据可能还在Insert Buffer中美也就是共享空间中，所以通过ibd文件进行恢复之后，还需要进行REPAIR TABLE操作来重建表上所有的辅助索引。

两次写(Double Write)：数据页的可靠性

发生写失效的情况，比如部分写失效，16KB的页只写了前4KB就宕机。重做日志中记录的是对页的物理操作，如偏移量800，写‘aaa’记录，也就是说如果这个页本身已经发生了损坏，再对其重做是没有意义的。所以在应用重做日志之前，用户需要一个页的副本，当写入失效发生时，先通过页的副本还原该页，再进行重做，这就是doublewrite。

doublewrite分为两部分，一部分是内存中的doublewrite buffer，大小2M，另一部分是磁盘上共享表空间的连续128个页，大小也是2M。对缓冲池脏页进行刷新时，先用memcpy函数将脏页复制到内存的doublewrite buffer，然后由doublewrite buffer再分为两次，每次1MB顺序写入共享表空间的物理磁盘上，然后马上调用fsync函数，同步磁盘。doublewrite页(共享表空间的页)是连续的，所以这是一个顺序的过程，并没有很大的开销。若操作系统在将页写入磁盘的过程中发生了崩溃，在恢复过程中，InnoDB存储引擎可以从共享表空间中的doublewrite中找到该页的一个副本，复制到表空间文件再应用重做日志。完成共享表空间相关的写入后，再将doublewrite buffer中的页写入各个表空间文件中，而此时的写入则是离散的，因为各个表空间可能在磁盘的不同位置。

自适应哈希索引(Adaptive Hash Index)

哈希查询的一般时间复杂度是O(1)，而生产环境下的B+树的查找次数一般为3～4次。自适应哈希索引是InnoDB的一个特殊功能，当InnoDB发现某些索引使用十分频繁的时候，它会在内存中基于B-Tree索引上再创建一个哈希索引，建立速度很快，而且不需要对整张表构造哈希索引。这样可以让B-Tree索引也具有哈希索引的一些优点，比如快速的哈希查找。自适应和希索引是一个完全自动的内部行为，用户无法控制或者配置，会自动根据访问的频率和模式来自动地为某些热点页建立哈希索引。但是有必要的话可以关闭该功能。当InnoDB存储引擎在对表上各索引页的查询进行监控时，观察到建立哈希索引可以带来速度提升时，则建立索引，称为AHI。

AHI有一个要求，即对这个页的连续访问模式必须是一样的。例如对于(a, b)这样的联合索引页，其访问模式可以时以下两种情况：

  • WHERE a=XXX
  • WHERE a=XXX and b=XXX

访问模式一样指的是查询的条件一样，若交替进行上述两种查询，那么InnoDB存储引擎不会对该页构造AHI。

启用AHI之后，读取和写入速度可以提高2倍，辅助索引的连接操作性能可以提高5倍，是一种很好的优化模式，且其设计思想是数据库自优化的，无需DBA进行人为调整。

异步IO(Async IO)：提高磁盘操作性能

用户可以在发出一个IO请求后立即再发出另一个IO请求，当全部IO请求发送完毕之后，等待所有IO操作的完成，这就是AIO。AIO还可以进行IO Merge操作，也就是将多个IO合并为1个IO，这样可以提高IOPS性能。

刷新邻接表(Flush Neighbor Page)

当刷新一个脏页时，InnoDB存储引擎会检测该页所在区(extend)的所有页，如果是脏页，那么一起进行刷新。

Oracle和MySQL

  • Oracle是大型数据库而MySQL是中小型数据库，Oracle不开源价格昂贵，Mysql开源。

  • SQL语法不同，MySQL比较灵活而Oracle比较严格，比如单引号使用，分页的处理以及主键的自动增长。

  • 关于事务，MySQL默认对事务不支持，但innodb存储引擎支持，Oracle对事务完全支持。

  • MySQL是单进程多线程，Oracle是多进程(Windows下也是单进程)，在Windows操作系统下只能使用一个用户，或者是在不同的用户之间做切换，但在Linux下，是多用户或者相同用户可以同时连接操作系统。

  • 数据库和实例以及用户之间的关系。

    MySQL的实例是用户登陆时系统分配给用户的，而用户必须是先在MySQL中创建好，然后登陆用户mysql -u username -p。instance>database

    Oracle的实例是在创建数据库的时候就默认创建好的，而用户基于数据库实例，实例之间可以没有关系所以其中的用户也不尽相同，登陆不同的实例相当于登陆了不同的数据库。

    instance=database

  • SQL语法

    • 建表时，Oracle不支持在建表的同时增加字段注释，需要单独加注释

      MySQL可以同时增加字段注释

    • 自动增长的数据类型处理

      MySQL有自动增长的数据类型，插入记录时不用操作此字段，会自动获得数据值。Oracle没有自动增长的数据类型，需要建立一个自动增长的序列号，插入记录时要把序列号的下一个赋值给该字段。

    • 单引号的处理

      MySQL中可以用双引号包含字符串，Oracle只能用单引号包起字符串。在插入和修改字符串之前必须要做单引号的替换：把所有出现的一个单引号替换成两个单引号。

    • 分页

      MySQL中用LIMIT开始位置，记录个数，Oracle中每个结果集只有一个ROWNUM字段标明它的位置。

  • 索引

    索引是关系数据库中用于存放每一条记录的一种对象，主要目的是加快数据的读取速度和完整性检查。建立索引是一项技术性要求高的工作，一般在数据库设计阶段与数据库结构一道考虑，应用系统的性能与索引的合理直接有关。

    Oracle和MySQL的常用索引结构基本都是B树。

MySQL索引

Posted on 2019-08-15

MySQL索引

索引类型

  • 普通索引

最基本的索引，没有任何限制

  • 唯一索引 unique

与普通索引类似，区别是：索引列的值必须唯一，但允许有空值。如果是组合索引，则列值的组合必须唯一。

  • 主键索引 primary

特殊的唯一索引，一个表只能有一个主键，不允许有空值。一般在建表的时候同时创建主键索引。

  • 组合索引

多个字段上创建的索引，只有在查询条件中使用了创建索引时的第一个字段，索引才会被使用。使用组合索引时遵循最左前缀集合。

  • 全文索引 fulltext (大量文本检索时)

主要用来查找文本中的关键字，而不是直接与索引中的值相比较。fulltext 与其他索引不大相同，更像是一个搜索引擎。不是简单的 where 语句的参数匹配。配合 match against 操作使用，并不是一般的 where 语句加 like。可以在 char、varchar、text 列上创建全文索引。

match() 函数中指定的列必须和全文索引中指定的列完全相同，否则就会报错，无法使用全文索引，这是因为全文索引不会记录关键字来自哪一列。如果想要对某一列使用全文索引，请单独为该列创建全文索引。索引的关键词用against标识,IN BOOLEAN MODE(布尔全文索引)是只有含有关键字就行,不用在乎位置,是不是起启位置.

  • 在数据量较大的时候，将数据放入一个没有全局索引的表中，然后再用create index 创建 fulltext 索引，要比先为一张表创建 fulltext 然后再将数据写入的速度快很多。
  • MySQL 中的全文索引，有两个变量，最小搜索长度和最大搜索长度，对于长度小于最小搜索长度和大于最大搜索长度的词语，都不会被索引。通俗点就是说，想对一个词语使用全文索引搜索，那么这个词语的长度必须在以上两个变量的区间内。show variables like ‘%ft%’;查看发现只能检索 3 或 4—84 区间范围内的关键字，当然可以修改这个大小。

语句

  • unique|fulltext为可选参数，分别表示唯一索引、全文索引
  • index和key为同义词，两者作用相同，用来指定创建索引
  • col_name为需要创建索引的字段列，该列必须从数据表中该定义的多个列中选择
  • index_name指定索引的名称，为可选参数，如果不指定，默认col_name为索引值
  • length为可选参数，表示索引的长度，只有字符串类型的字段才能指定索引长度
  • asc或desc指定升序或降序的索引值存储

基本上所有的索引都有三种创建方式：直接用语句创建索引，修改表结构创建索引和创建表的时候直接指定，一般情况下主键索引会直接指定。

索引实现

为什么不用Hash索引？

  • 只能满足等值查询，不能使用范围查询。
  • 无法被用来避免数据的排序操作。哈希索引是利用哈希值排序的，这个排序跟真实数据的排序是不一样的，无法利用索引排序。
  • 不能利用部分索引键查询。哈希索引的联合查询是对合并之后的数据的值的哈希值进行索引。
  • 不能避免表扫描。不同的键值可能会有相同的哈希值。
  • 大量Hash值冲突时并不一定会比B树效率高。

  • MySQL索引的实现在存储引擎层而不是服务器层。

  • MySQL 索引底层实现是 B+Tree，B+Tree 对索引列顺序存储，很适合查找范围数据。

    > BTree 对索引有所限制
    • 必须要遵循最左原则，否则无法使用索引
      • 不能跳过索引中的列
      • 如果查询中有某列的范围查询，则其右边所有列都无法使用索引优化查询。

  • 哈希索引基于哈希表实现，要求查询要精确匹配所有的列。哈希索引的原理是：对于每一行数据，对所有的索引列都会计算一个哈希值，不同键值的行计算出来的哈希值都不一样。哈希索引将所有的哈希值存储在索引中，同时保存指向每个数据行的指针，这样就可以根据哈希值找到指针并返回到行。

  • 哈希索引的限制：(但若是可以使用哈希索引，效率会很高)

    • 只包含哈希值和指针，不包含字段值，所以无法使用”覆盖索引“这一优化方式。

      覆盖索引：就是 select 的数据列只用从索引中就可以取得，不必从数据表中读取，也就是说查询列要被所使用的的索引覆盖。只有 B-Tree 索引可以做覆盖索引，因为要存储索引的列。

    • 并不是按照顺序存储的，索引不能用于排序。

    • 不支持部分索引列匹配查找，哈希索引始终是使用索引列的全部内容计算哈希值的。
    • 只支持等值查询，不支持范围查询。

Innodb 有一个特殊的功能：当注意到一些索引值被使用的非常频繁并且符合哈希特点(每次查询的列都一样)，会在内存中基于 B-Tree 的索引之上再创建一个哈希索引，完全自动的内部行为。

聚簇索引和非聚簇索引

密集索引、稀疏索引、聚集索引、聚簇索引？？？？聚集索引和聚簇索引应该是一个东西

  • 密集索引：文件中的每一个搜索码值都对应一个索引值(每一个索引键值都对应有一个索引项)

  • 稀疏索引：文件只为索引码的某些值建立索引(相对于稠密索引，稀疏索引只为某些搜索码值建立索引记录；在搜索时，找到其最大的搜索码值小于或等于所查找记录的搜索码值的索引项，然后从该记录开始向后顺序查询直到找到为止。 )

  • 聚集索引：指索引项的排列方式和表中数据记录排序方式一致的索引。也就是说聚集索引的顺序就是数据的物理存储顺序，它会根据聚集索引键的顺序来存储表中的数据，即对表的数据按索引键的顺序进行排序，然后重新存储到磁盘上。因为数据在物理存放时只能有一种排列方式，所以一个表只能有一个聚集索引。(主键是一种聚集索引)？？？

  • 聚簇索引：术语“聚族”表示数据行和相邻的键值紧凑的存储在一起。因为无法同时把数据行放在两个不同的地方，所以一个表只能有一个聚族索引。

  • myisam对应非聚簇索引，mysql对应聚簇索引

myisam在用到索引时先到索引表(树)中查找，找到数据所在的行，再去数据表中拿到数据。所以 myisam 的索引文件和数据文件是分开的，叫做非聚簇索引。索引指向数据在行的位置。索引和数据分开存储。

而在聚簇索引中，索引文件和数据文件是同一个文件，所以在 innodb 中找到了索引也就找到了数据，既存储主键索引值，又存储行数据，称为聚簇索引。innodb 索引指向主键对数据的引用，非主键索引则指向对主键的引用，若没有主键索引，会使用唯一索引，若没有唯一索引则会使用数据库内部的一个行的 id来当做主键索引。

并不是一种单独的索引类型，而是一种数据存储方式。一般情况下主键是聚簇索引的索引列。

innodb 的聚簇索引在同一个 B-Tree 中保存了索引列和具体的数据，实际的数据保存在叶子页中，中间的节点页保存指向下一层页面的指针。“聚簇”的意思是数据行被按照一定顺序一个个紧密的排列在一起存储，一个表只能有一个聚簇索引，因为数据的存放方式只有一种。

  • 在聚簇索引中，数据会被按照顺序整理排序，所以使用 where 进行范围查询时，会大大加快速度，但非聚簇索引不会对数据进行排序。
  • 聚簇索引将主键组织到一颗B+树上，行数据储存在叶子节点上，在查询主键时，可以直接通过B+树的检索算法找到对应的叶节点，找到行数据；但是在对非主键数据进行查找的时候，需要第一步先在辅助索引中找到主键，再用主键执行一遍 B+ 树的搜索算法找到行数据

在 innodb 中聚簇索引就是表，不需要像 myisam 那样建立独立的行存储。

myisam 会将数据按照插入的顺序存储在磁盘上，创建索引时二级索引的叶子节点包含行号和主键或者索引列，会按照索引列排序。

聚簇索引的优势
  • 行数据和叶子节点存储在一起，同一页中会有多行数据，访问同一数据页不同行记录时，已经将页加载到了 buffer 中，再次访问的时候，会在内存中完成访问，不必访问磁盘。主键和行数据一起被载入内存，找到叶子节点就可以返回行数据。
  • 辅助索引使用主键作为“指针”而不是使用地址值作为指针的好处是减少了当出现行移动或者数据页分裂时辅助索引的维护工作。
  • 聚簇索引的数据的物理存放顺序与索引顺序是一致的，即：只要索引是相邻的，那么对应的数据一定也是相邻的存放在磁盘上的。
  • 非聚簇索引的物理地址是凌乱的，拿到这些物理地址，按照合适的算法进行 I/O 读取，会不停的寻道，聚簇索引只需要一次。

维护表的目的是找到并修复损坏的表，维护准确的索引统计信息，减少碎片。
